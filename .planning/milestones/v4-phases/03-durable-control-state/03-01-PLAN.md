<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin/API input -> controlstate validation | Untrusted provider configuration will eventually cross into durable state. |
| controlstate repository -> database | Durable records include encrypted provider credentials and must never seed user configuration silently. |
| encrypted storage -> runtime decrypt | Plaintext secrets are only allowed at adapter construction time in later plans. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | Information Disclosure | `provider_secrets` and `SecretCipher` | mitigate | Store only AES-GCM ciphertext/nonce in durable storage; tests assert raw secrets and ciphertext are absent from redacted records, validation errors, and logs per D-10 and D-29. |
| T-03-01-02 | Tampering | migrations | mitigate | Schema-only migrations create tables and `schema_migrations`; they contain no provider/API key seed inserts per D-01 and D-02. |
| T-03-01-03 | Elevation of Privilege | backend capabilities | mitigate | `CapabilityProfile` explicitly rejects unsupported distributed/future features for SQLite and rejects semantic cache/rate/cost features in Phase 3 per D-06 and D-32. |
| T-03-01-04 | Repudiation | future Admin changes | mitigate | `audit_events` schema captures actor label, timestamp, target provider ID, action, outcome, and redacted metadata per D-21. |
