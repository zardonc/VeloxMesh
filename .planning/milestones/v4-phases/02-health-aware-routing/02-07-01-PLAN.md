# Phase 2.7 Plan 1: Adapter Capability Contract and Registry Metadata

<objective>
Add the provider-neutral capability contract at the adapter and registry boundary.

This plan is complete when all provider adapters implement a shared capability metadata contract, the registry can list and look up that metadata in stable provider order, and all compile-impact tests/mocks are updated.
</objective>

<context>
Use `.planning/phases/02-health-aware-routing/02-07-CONTEXT.md` as the source of Phase 2.7 decisions.

Important locked decisions:
- D-01 through D-04: capabilities describe adapter support without enabling new behavior, and SDK-native details stay inside adapter packages.
- D-05 through D-12: capability metadata covers provider type, `chat_completions`, text input/output, false streaming/tools flags, generation parameters, and optional static constraints if already known.
- D-13 through D-16: common capability types belong in `internal/providers`, with registry APIs and copy-safe metadata.
- D-20 through D-22: tests must prove current adapters report conservative metadata and false values for unimplemented features.
</context>

<scope>
## Must Build

- Provider-neutral capability structs/types and constants in `internal/providers`.
- Adapter contract method for capability reporting.
- Capability reporting for OpenAI-compatible, Anthropic, and Gemini adapters.
- Registry APIs for provider capability lookup/listing.
- Copy-safe returned metadata.
- Test updates for all existing `ProviderAdapter` mocks affected by the new method.
- Unit tests for adapter and registry capability metadata.

## Must Not Build

- Readiness or `/v1/models` capability exposure; that is Plan 2.
- Streaming implementation.
- Tool/function calling implementation.
- Multimodal implementation.
- Live model discovery.
- Admin API or Admin Console.
- Redis, PostgreSQL, runtime CRUD, rate limiting, semantic cache, or cost governance.
</scope>

<truths>
- Current adapters should report implemented behavior only: non-streaming text chat completions with supported `temperature` and `max_tokens` mappings.
- Capability metadata must be deterministic and secret-safe.
- Any slices/maps returned from capability APIs must be copies.
- Updating `ProviderAdapter` affects mocks in provider, routing, gateway, and health tests.
</truths>

<tasks>

## 1. Define Provider Capability Types

type: implementation

files:
- `internal/providers/capabilities.go`
- `internal/providers/adapter.go`

action:
- Create `internal/providers/capabilities.go`.
- Define provider-neutral typed constants for:
  - provider types: `openai-compatible`, `anthropic`, `gemini`.
  - operations: `chat_completions`.
  - modalities: `text`.
  - generation parameters: `temperature`, `max_tokens`.
- Define a `CapabilitySet` struct with:
  - provider type.
  - supported operations.
  - input modalities.
  - output modalities.
  - `Streaming` bool.
  - `ToolCalling` bool.
  - generation parameter support.
  - optional constraints for future static limits if needed, but do not add config parsing in this plan unless a zero-value struct is useful.
- Add `Clone` or copy helpers for `CapabilitySet`.
- Extend `ProviderAdapter` with `Capabilities() CapabilitySet`.

verify:
- `go test ./internal/providers` should compile after mocks in Task 3 are updated.

acceptance_criteria:
- Capability types live in `internal/providers`.
- The adapter interface requires provider-neutral capability reporting.
- No provider-native SDK types appear in capability structs.

acceptance_criteria:
- Registry callers can inspect current provider capabilities without provider-specific imports.
- Current adapters report only supported text chat capabilities.
