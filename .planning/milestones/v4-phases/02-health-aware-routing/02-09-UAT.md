## Tests

### 1. Model Catalog Tracks Provider Eligibility
expected: The gateway has an internal copy-safe model catalog that maps model IDs to eligible providers using configured adapter models and provider capabilities. Shared model names retain multiple eligible providers internally, while provider-specific model names remain eligible only for the provider that configured them.
result: pass

### 2. Routing Rejects Unknown and Ineligible Models
expected: Chat requests for unknown models are rejected before provider execution with `ErrNoEligibleProvider`, and strict `X-Route-To` requests to providers that do not support the requested model/operation are rejected with `ErrIneligibleProviderOverride`.
result: pass

### 3. Fallback Skips Healthy But Ineligible Providers
expected: During fallback, the gateway excludes previously failed providers and also ignores healthy providers that are not eligible for the requested model, so provider-specific model requests are never sent to the wrong provider.
result: pass

### 4. Public Models API Remains OpenAI-Compatible
expected: `/v1/models` still returns the OpenAI-compatible `object: "list"` response with deduplicated model items and does not expose internal provider IDs, capability metadata, API keys, prompts, or provider-native payload details.
result: pass
