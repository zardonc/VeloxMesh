# Phase 2 Plan: Health-Aware Multi-Provider Routing

<objective>
Extend the Phase 1 Go gateway from a single-provider OpenAI-compatible proxy into a health-aware multi-provider routing layer.

The phase is complete when VeloxMesh can be configured with multiple OpenAI-compatible providers, select providers via round-robin or least-latency routing, avoid unhealthy providers, update in-memory provider health after requests, expose readiness/model aggregation accurately, and prove all behavior with unit and integration tests using fake upstream providers.
</objective>

<context>
Use `.planning/phases/02-health-aware-routing/02-CONTEXT.md` as the source of phase decisions.

Important locked decisions:

- Keep the gateway Go 1.22+ and Chi-based.
- Keep providers OpenAI-compatible in Phase 2.
- Do not add PostgreSQL, Redis, Admin API, streaming, semantic cache, rate limiting, or cost governance.
- Add routing value through multi-provider config, health state, and lightweight routing strategies.
- Before implementing provider-specific behavior, check official SDK/reference behavior. Use a full SDK only if it is lightweight and reduces correctness risk; otherwise extract/reimplement only the minimal needed behavior inside the adapter.
</context>

<scope>
## Must Build

- Multi-provider static configuration.
- Provider registry capable of storing and listing multiple adapters.
- In-memory concurrency-safe provider health store.
- Health-aware `round-robin`, `least-latency`, and `override` routing.
- Gateway service health updates around provider calls.
- `/readyz` provider readiness summary.
- `/v1/models` model aggregation across providers.
- Structured errors for no healthy provider, unknown provider override, and unhealthy provider override.
- Logs/metrics hooks for selected provider, strategy, health status, latency, pending, and outcome.
- Tests with at least two fake OpenAI-compatible upstream providers.

## Must Not Build

- PostgreSQL provider table.
- Redis health state.
- Admin API provider CRUD.
- Circuit breaker state machine.
- Retries or fallback-chain execution.
- Composite-score routing.
- YAML heuristic rules.
- Native Anthropic/Gemini/Ollama adapters.
- SSE streaming.
- Semantic cache.
- Prometheus or OpenTelemetry exporters beyond existing metric abstraction.
</scope>

<truths>
- Provider health is in-memory for Phase 2 but must sit behind an interface so Redis can replace or back it later.
- Unhealthy providers must not receive traffic through normal routing.
- `X-Route-To` must remain supported, but should not force an unhealthy provider.
- Pending counts must be decremented on every provider call path, including errors and context cancellation.
- Tests should not depend on real external LLM providers.
- Do not log API keys, raw prompts, or authorization headers.
</truths>

## 4. Add In-Memory Provider Health Store

type: implementation

files:
- `internal/health/store.go`
- `internal/health/store_test.go`
- `internal/providers/adapter.go`

action:
- Create `internal/health`.
- Define provider health status values:
  - `healthy`
  - `degraded`
  - `unhealthy`.
- Define a concurrency-safe `Store` interface:
  - `EnsureProvider(id string)`
  - `BeginRequest(id string)`
  - `EndRequest(id string, latency time.Duration, err error)`
  - `Snapshot(id string)`
  - `Snapshots()`
  - optional `SetStatus(id, status, reason)`.
- Track:
  - EWMA latency
  - pending count
  - consecutive failures
  - total successes/failures
  - last error
  - last updated timestamp.
- Use simple configurable thresholds:
  - healthy: no recent/consecutive failure
  - degraded: failures below unhealthy threshold
  - unhealthy: consecutive failures >= threshold.
- On success, reduce or reset consecutive failures and allow recovery.

verify:
- Unit-test pending increments/decrements.
- Unit-test EWMA updates on success.
- Unit-test degraded/unhealthy transitions.
- Unit-test recovery after success.
- Unit-test concurrent `BeginRequest` / `EndRequest` does not race.

acceptance_criteria:
- Store has no external dependencies.
- Store can later be backed by Redis without changing routing/gateway call sites.

## 5. Implement Health-Aware Routing Strategies

type: implementation

files:
- `internal/routing/router.go`
- `internal/routing/router_test.go`
- `internal/errors/errors.go`

action:
- Replace or extend `StaticRouter` with a health-aware router.
- Support configured strategy:
  - `round-robin`: rotate across healthy/degraded providers.
  - `least-latency`: pick lowest EWMA latency provider when samples exist.
  - cold start: fall back to round-robin or default provider until latency samples exist.
  - `override`: honor `X-Route-To` for known healthy/degraded providers.
- Exclude unhealthy providers from normal routing.
- Reject unknown route override with a structured routing error.
- Reject unhealthy route override with a structured routing error.
- If no providers are healthy/degraded, return a structured no-healthy-provider error.
- Include `ProviderID`, `Strategy`, and health metadata in `RoutingDecision`.

verify:
- Test round-robin distributes across providers.
- Test least-latency picks lower EWMA provider.
- Test cold-start fallback.
- Test unhealthy provider is skipped.
- Test all-unhealthy returns no healthy provider error.
- Test `X-Route-To` known healthy provider succeeds.
- Test `X-Route-To` unhealthy provider fails.

acceptance_criteria:
- Handler code does not choose providers.
- Router is deterministic enough for reliable unit tests.
- Routing behavior matches Phase 2 context decisions.
