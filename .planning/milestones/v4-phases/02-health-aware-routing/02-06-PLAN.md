# Phase 2.6 Plan: Active Provider Health Probing and Recovery

<objective>
Add configurable in-process provider health checks so provider health can recover without waiting for live client traffic.

The phase is complete when the gateway can periodically call cheap adapter health checks, update provider health and probe metadata, recover an unhealthy provider after successful probes, expose secret-safe readiness details, and provide internal `ProbeOnce(ctx)` / `ProbeProvider(ctx, providerID)` boundaries suitable for a future Admin Console/Admin API.
</objective>

<context>
Use `.planning/phases/02-health-aware-routing/02-06-CONTEXT.md` as the source of phase decisions.

Important locked decisions:
- D-01 through D-04: This phase is active health probing and recovery, not a broad observability/platform-management phase.
- D-05 through D-09: Add a lightweight prober that calls `ProviderAdapter.HealthCheck(ctx)` and complements live request-path health updates.
- D-10 through D-14: Health-check parameters must be structured config so future Admin Console/Admin API callers can reuse the same prober boundary.
- D-15 through D-18: Preserve simple `healthy` / `degraded` / `unhealthy` semantics and avoid a full circuit breaker state machine.
- D-19 through D-22: `/readyz` should expose secret-safe probe visibility and still base readiness on routeable providers.
- D-23 through D-25: Prober lifecycle must be context-bound and testable without real LLM calls.
</context>

<scope>
## Must Build

- Structured health-check config with conservative defaults and validation.
- Optional per-provider health-check overrides for fields needed now.
- In-process health prober with deterministic `ProbeOnce(ctx)` and `ProbeProvider(ctx, providerID)`.
- Context-driven background loop for runtime probing.
- Health-store support for probe result metadata and probe-driven recovery.
- App wiring that starts the prober without leaking goroutines in tests.
- `/readyz` updates that include secret-safe probe state/freshness when available.
- Unit and integration tests for config, prober behavior, recovery, readiness, and routing after recovery.
- README config example update.

## Must Not Build

- Admin Console UI.
- Admin API endpoint.
- Redis-backed health state.
- PostgreSQL-backed provider/config state.
- Runtime health policy CRUD or hot reload.
- Full Closed/Open/HalfOpen circuit breaker state machine.
- Synthetic chat/model generation probes.
- Prometheus/OpenTelemetry exporters.
- Streaming retry, hedged requests, rate limiting, semantic cache, or cost governance.
</scope>

<truths>
- Health checks must be cheap and must not call generation/model-completion endpoints.
- Health check config must be represented as named structs, not scattered loose fields.
- `ProbeProvider(ctx, providerID)` is part of the internal contract for later Admin Console/Admin API use, but this phase does not expose an HTTP admin endpoint.
- Successful probe recovery must make an unhealthy provider routeable again through the existing router.
- Existing request-path health updates from Phase 2.5 must remain intact.
- No API keys, auth headers, raw prompts, raw upstream bodies, or full provider error payloads may be exposed in readiness, logs, tests, or docs.
- Tests must use fake adapters or fake upstream servers and must not call real providers.
</truths>

## 2. Extend Health Store for Probe Results

type: implementation

files:
- `internal/health/store.go`
- `internal/health/store_test.go`

action:
- Extend `ProviderSnapshot` with secret-safe probe metadata if needed:
  - `LastProbeAt time.Time`
  - `LastProbeSuccess bool`
  - `LastProbeError string` or equivalent sanitized string.
  - `LastProbeDuration time.Duration`
- Add a method to the `Store` interface for probe results, for example:
  - `RecordProbe(id string, latency time.Duration, err error)`
  - or `EndProbe(id string, latency time.Duration, status providers.HealthStatus)`.
- Implement probe success as provider recovery:
  - reset consecutive failures according to configured success threshold.
  - clear last error when recovered.
  - optionally update EWMA latency with probe latency only if that does not distort request latency. If uncertain, keep probe latency in probe metadata rather than EWMA.
- Implement probe failure as provider failure:
  - increment consecutive failures and total failures or a probe-specific counter.
  - set status to degraded/unhealthy through existing threshold logic.
- Keep `BeginRequest` / `EndRequest` behavior unchanged.
- Ensure all changes are concurrency-safe.

verify:
- Unit-test probe success on healthy provider.
- Unit-test probe failure transitions provider to degraded/unhealthy based on threshold.
- Unit-test successful probe recovers an unhealthy provider.
- Unit-test request pending counts are not affected by probes.
- Unit-test probe metadata appears in snapshots and is sanitized.

acceptance_criteria:
- Probe results can drive health transitions without faking client requests.
- Existing request-path tests continue to pass.

## 3. Implement Provider Health Prober

type: implementation

files:
- `internal/health/prober.go`
- `internal/health/prober_test.go`
- `internal/providers/registry.go`

action:
- Add a prober type that receives:
  - provider registry or provider list.
  - health store.
  - resolved health-check config.
  - optional logger/metrics hooks if already easy to pass.
- Add deterministic methods:
  - `ProbeOnce(ctx context.Context)`.
  - `ProbeProvider(ctx context.Context, providerID string)`.
- Add runtime loop:
  - `Start(ctx context.Context)` or equivalent.
  - honor `enabled`, `initial_delay`, `interval`, `timeout`, and `max_concurrency`.
  - stop cleanly on context cancellation.
- Ensure `ProbeProvider` returns structured/sanitized result data useful to a future Admin Console, for example provider ID, available, message/category, duration, and timestamp.
- Call `adapter.HealthCheck(ctx)` with a timeout.
- Convert unavailable health statuses into sanitized errors or store results.
- Do not call `adapter.Complete` or any model generation endpoint.
- Add or reuse registry APIs needed to fetch one adapter by ID and list all adapters deterministically.

verify:
- Unit-test `ProbeProvider` success.
- Unit-test unknown provider.
- Unit-test health-check unavailable result records failure.
- Unit-test adapter timeout/cancellation.
- Unit-test `ProbeOnce` probes all configured providers.
- Unit-test background loop exits when context is canceled without goroutine leaks.
- Unit-test max concurrency with fake blocking adapters if practical.

acceptance_criteria:
- Probe logic is deterministic in tests and reusable by future admin endpoints.
- Runtime loop does not leak goroutines.
