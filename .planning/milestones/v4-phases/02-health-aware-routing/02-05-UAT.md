## Tests

### 1. Retryability Policy
expected: |
  Retryability is centralized and based on Phase 2.4 provider error categories.
result: pass
evidence: |
  `internal/errors/errors.go` defines `IsRetryableProviderError`. `internal/errors/errors_test.go` covers retryable categories (`provider_rate_limit`, `provider_timeout`, `provider_unavailable`, `provider_bad_response`, `provider_error`) and non-retryable categories (`provider_invalid_request`, `provider_invalid_model`, `provider_auth_error`, routing errors, context errors).

### 2. Static Fallback Config
expected: |
  Fallback config supports conservative defaults: enabled for multi-provider config and bounded `max_attempts`.
result: pass
evidence: |
  `internal/config/config.go` parses `fallback_enabled` and `max_attempts`, defaults fallback based on provider count, defaults max attempts to 2, and clamps attempts to 1..5. `internal/config/config_test.go` covers default enablement and max attempt validation.

### 3. Exclusion-Aware Routing
expected: |
  Router can select providers while excluding already-attempted providers and still respects provider health.
result: pass
evidence: |
  `internal/routing/router.go` adds `SelectExcluding`. `internal/routing/router_test.go` covers excluded providers, all-excluded behavior, and exclusion combined with unhealthy providers.

### 4. Fallback Success
expected: |
  A non-streaming chat request can recover from a retryable provider failure by falling back to another eligible provider and returning OpenAI-compatible success.
result: pass
evidence: |
  `tests/integration/chat_test.go` includes "Fallback Success": p1 returns 500, p2 succeeds, response status is 200, `X-Provider` is p2, `X-Fallback-Used` is true, and `X-Provider-Attempts` is 2.

### 5. Strict Route Override
expected: |
  `X-Route-To` remains strict and does not silently fallback to another provider.
result: pass
evidence: |
  `internal/gateway/service.go` sets max attempts to 1 when `RouteOverride` is present. `tests/integration/chat_test.go` includes "Strict Override No Fallback".

### 6. Non-Retryable Error Behavior
expected: |
  Non-retryable provider errors fail fast and do not mask bad client input or provider configuration.
result: pass
evidence: |
  `tests/integration/chat_test.go` includes "Non-Retryable Error (400) No Fallback" and "Non-Retryable Error (401 Auth) No Fallback".

### 7. Rate-Limit Fallback
expected: |
  `provider_rate_limit` / upstream 429 is retryable and can fallback to another provider.
result: pass
evidence: |
  `tests/integration/chat_test.go` includes "Rate Limit (429) Fallback Success", verifying a 429 upstream properly falls back to the next eligible provider.

### 8. Per-Attempt Health Updates
expected: |
  Every attempted provider gets health updates: failed providers record failures and successful fallback provider records success.
result: pass
evidence: |
  `internal/gateway/service_test.go` asserts that the gateway attempt loop records one provider failure and one successful provider path after fallback, regardless of round-robin selection order.

### 9. Attempt Metadata Headers
expected: |
  Successful responses expose safe fallback metadata without changing OpenAI-compatible response bodies.
result: pass
evidence: |
  `internal/http/handlers/chat.go` sets `X-Provider-Attempts` and `X-Fallback-Used`. Integration fallback success asserts both headers. Response body still uses `llm.ChatCompletionResponse`.

### 10. Attempt Metrics
expected: |
  Metrics/logging record provider attempts and final outcomes without leaking secrets.
result: pass

Dedicated `RecordProviderAttempt` / `RecordFallback` hooks were not added, but current behavior satisfies per-attempt outcome recording at the existing abstraction level.
