# Phase 2.4 Plan: Provider Reliability and Error Contract

<objective>
Harden the provider layer now that OpenAI-compatible, Anthropic, and Gemini adapters exist.

The phase is complete when all provider adapters return a shared structured provider error contract, the OpenAI-compatible adapter forwards the same common generation parameters as native adapters, malformed upstream responses become controlled gateway errors, provider health distinguishes provider failures from client/request failures where practical, readiness exposes clear routability without expensive model calls, and all behavior is covered by deterministic tests.
</objective>

<context>
Use `.planning/phases/02-health-aware-routing/02-04-CONTEXT.md` as the source of phase decisions.

Important locked decisions:
- D-01: This is a provider reliability and contract-hardening phase, not a new feature-surface phase.
- D-02: The main deliverable is a stable provider error/health contract across OpenAI-compatible, Anthropic, and Gemini adapters.
- D-03: Do not add Redis, PostgreSQL, Admin API, streaming, retries, fallback chains, semantic cache, rate limiting, cost governance, Prometheus exporters, or OpenTelemetry exporters.
- D-04 through D-07: All adapters should use shared provider error categories, and handlers should remain provider-agnostic.
- D-08 through D-12: Health and readiness should distinguish provider/config failures from client/request failures where practical, while staying lightweight and in-memory.
- D-13 through D-16: All adapters should honor `Model`, `Messages`, `Temperature`, `MaxTokens`, and normalize successful responses back to OpenAI-compatible output.
- D-17 through D-19: Logs/metrics should include useful provider diagnostics without API keys, auth headers, raw prompts, or raw provider response bodies.
</context>

<scope>
## Must Build

- Shared provider error taxonomy and helpers for adapter, handler, health, and routing use.
- Structured provider error mapping in the OpenAI-compatible adapter.
- Consistent category usage in Anthropic and Gemini adapters.
- `Temperature` and `MaxTokens` forwarding in the OpenAI-compatible adapter.
- Controlled `provider_bad_response` errors for empty, malformed, or unusable upstream responses.
- Provider health impact rules that avoid poisoning provider health for client/request errors.
- Lightweight readiness behavior that reflects configured provider routability and adapter health semantics where available.
- Secret-safe logs/metrics fields for provider id, provider type, model, strategy, error category, health status, and latency.
- Unit and integration tests for the reliability contract.

## Must Not Build

- Streaming, tool/function calling, multimodal, embeddings, Ollama, or generic HTTP adapter support.
- Retry/fallback-chain execution or circuit breaker state machines.
- Redis-backed health, PostgreSQL provider CRUD, or Admin API.
- Rate limiting, admission queues, semantic cache, cost accounting, or token aggregation.
- Prometheus/OpenTelemetry exporters beyond the existing internal observability abstraction.
</scope>

<truths>
- `/v1/chat/completions` successful responses must remain OpenAI-compatible.
- HTTP handlers should not need provider-specific error branches.
- Provider adapters may use official SDK behavior where it reduces correctness risk; Anthropic and Gemini should continue using their official SDKs.
- OpenAI-compatible provider support should remain a small local HTTP adapter unless a concrete SDK adoption reason emerges.
- Provider errors must not expose API keys, auth headers, raw prompts, or raw provider response bodies.
- Tests must use fake upstreams or SDK test transports and must not call real external LLM providers.
- Health state remains in-memory for this phase.
</truths>

<tasks>

## 1. Define Shared Provider Error Contract

type: implementation

files:
- `internal/errors/errors.go`
- `internal/providers/adapter.go`
- `internal/gateway/service.go`
- `internal/http/handlers/chat.go`

action:
- Add a shared provider error contract using the existing `errors.GatewayError` envelope if it remains the cleanest import boundary, or introduce a small provider error type if that avoids cycles.
- Support these stable categories:
  - `provider_auth_error`
  - `provider_rate_limit`
  - `provider_invalid_request`
  - `provider_invalid_model`
  - `provider_timeout`
  - `provider_unavailable`
  - `provider_bad_response`
  - `provider_error`
- Ensure each category has a deterministic HTTP status mapping:
  - auth/config failures: 502 or 503, depending on existing handler convention.
  - rate limit: 429 or 502, but choose one consistent gateway-facing rule and test it.
  - invalid client request: 400.
  - invalid model: 400 or 404, matching current API semantics.
  - timeout/unavailable/provider generic failures: 502 or 503 according to routability impact.
  - bad upstream response: 502.
- Include provider metadata only when safe and useful:
  - provider id
  - provider type
  - model
  - category/code
  - retryability or health impact if useful to avoid ad hoc string checks.
- Add helper functions for health handling, for example:
  - provider/auth/config errors affect health.
  - timeout/unavailable/provider generic/bad response affect health.
  - invalid request should not affect provider health when caused by client input.
  - invalid model should affect health only if it indicates provider config/model mapping is wrong.
- Keep `chat.go` provider-agnostic by mapping the structured error envelope rather than branching on provider implementations.

verify:
- Unit tests cover category construction and HTTP status mapping.
- Unit tests cover health-impact helper behavior.
- Existing routing errors (`ErrNoHealthyProvider`, `ErrUnknownProviderOverride`, `ErrUnhealthyProviderOverride`) still map correctly.

acceptance_criteria:
- All provider adapters can return one shared error shape.
- Handler and gateway code can reason about provider failures without provider-specific type switches.
- No test requires matching raw upstream error bodies.

## 2. Harden OpenAI-Compatible Adapter

type: implementation

files:
- `internal/providers/openai/adapter.go`
- `internal/providers/openai/adapter_test.go`

action:
- Forward common `llm.LLMRequest` fields into the outbound OpenAI-compatible JSON body:
  - `model`
  - `messages`
  - `temperature` when non-nil
  - `max_tokens` when non-nil.
- Preserve existing auth and `/chat/completions` behavior.
- Replace plain `fmt.Errorf` provider failures with structured provider errors:
  - 400-style upstream errors -> `provider_invalid_request`, unless response shape clearly indicates invalid model.
  - 401/403 -> `provider_auth_error`.
  - 404 or provider-specific invalid-model payloads -> `provider_invalid_model` when detectable.
  - 408/context timeout/net timeout -> `provider_timeout`.
  - 429 -> `provider_rate_limit`.
  - 5xx/connectivity/refused/DNS-like failures -> `provider_unavailable` or `provider_error` using deterministic rules.
  - decode failures, empty choices, missing message content, or malformed JSON -> `provider_bad_response`.
- When reading non-2xx responses, bound the response body read and sanitize details before attaching messages.
- Do not log or return raw provider response bodies that may include prompts or secrets.

verify:
- Test `temperature` and `max_tokens` forwarding with an `httptest.Server`.
- Test each major status-code mapping.
- Test context timeout or net timeout mapping if practical without timing flakes.
- Test malformed JSON maps to `provider_bad_response`.
- Test `choices: []` or missing assistant content maps to `provider_bad_response`.

acceptance_criteria:
- OpenAI-compatible adapter is no longer the least-structured adapter.
- OpenAI-compatible behavior matches the same reliability contract as native providers.

## 4. Refine Gateway Health Impact Semantics

type: implementation

files:

- `internal/health/store.go`
- `internal/health/store_test.go`
- `internal/errors/errors.go`

action:
- Update gateway service health recording to use structured provider error metadata.
- Ensure provider calls still always decrement pending counts on success, provider error, context cancellation, and handler cancellation.
- Apply health impact rules consistently:
  - provider auth/config/unavailable/timeout/bad-response/generic provider errors count as provider failures.

acceptance_criteria:

## 5. Improve Readiness Without Expensive Provider Calls

type: implementation

files:
- `internal/providers/adapter.go`
- `internal/http/handlers/health.go`
- `internal/gateway/service.go`
- `tests/integration/health_test.go`

action:
- Review the existing `ProviderAdapter.HealthCheck` contract and current readiness handler behavior.
- If adapters already expose lightweight `HealthCheck`, incorporate it into readiness only when it does not require a model generation call.
- If adapter health checks would require expensive upstream calls, keep readiness based on in-memory health snapshots and configured provider metadata, and document that decision in code comments/tests.
- `/readyz` should clearly indicate:
  - overall readiness.
  - configured provider count.
  - healthy/degraded/unhealthy counts.
  - provider id and provider type where already exposed.
  - whether at least one provider is routable.
- Return 200 when at least one provider is routable and 503 when none are routable.
- Keep readiness output secret-safe and avoid raw provider details.

verify:
- Integration tests cover all healthy, mixed healthy/degraded/unhealthy, and no routable provider states.
- Tests confirm readiness does not call fake model-generation endpoints just to report status.

acceptance_criteria:
- Readiness reflects provider routability better than a static process liveness check.
- Readiness stays lightweight and safe for frequent probes.
