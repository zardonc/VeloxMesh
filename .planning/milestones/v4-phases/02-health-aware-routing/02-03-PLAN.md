# Phase 2.3 Plan: Native Anthropic and Gemini Provider Adapters

<objective>
Extend the gateway to support native Anthropic and Gemini models via provider-specific adapters, translating OpenAI-compatible incoming requests into native SDK calls, and normalizing their responses back into OpenAI-compatible outputs.
</objective>

<context>
Use `.planning/phases/02-health-aware-routing/02-03-CONTEXT.md` as the source of phase decisions.

Important locked decisions:
- Use official SDKs: `github.com/anthropics/anthropic-sdk-go` and `google.golang.org/genai` (if practical).
- The HTTP handler (`/v1/chat/completions`) must remain strictly OpenAI-compatible.
- `llm.LLMRequest` must be extended to forward `Temperature` and `MaxTokens`.
- Adapters are responsible for specific error mapping (auth, rate limits, invalid models) to gateway errors without leaking raw details.
- No streaming, multimodal, or tool calling in this phase.
</context>

<scope>
## Must Build
- Extend `llm.LLMRequest` with `Temperature` and `MaxTokens`.
- `anthropic` and `gemini` provider config types.
- Native Anthropic adapter using the official Go SDK.
- Native Gemini adapter using the official Go SDK.
- OpenAI -> Native request translation.
- Native -> OpenAI response normalization (especially mapping finish reasons).
- App wiring to instantiate new adapters.
- Unit and Integration tests.

## Must Not Build
- Streaming, Tool Calling, Multimodal support.
- Redis/Postgres storage.
- Cost accounting.
</scope>

## 2. Implement Anthropic Native Adapter

type: implementation

files:
- `internal/providers/anthropic/adapter.go`
- `internal/providers/anthropic/adapter_test.go`

action:
- Import `github.com/anthropics/anthropic-sdk-go`.
- Implement `providers.ProviderAdapter`.
- Map OpenAI `system` messages into Anthropic `system` param. Map `user` and `assistant` messages to Anthropic Messages API format.
- Forward `Temperature` and `MaxTokens` (defaulting `MaxTokens` to 4096 if absent, as required by Anthropic).
- Map Anthropic response to `llm.LLMResponse`. 
  - Finish reason mapping: `end_turn` -> `stop`, `max_tokens` -> `length`, `stop_sequence` -> `stop`.
- Map SDK errors to structured gateway errors.

verify:
- Unit tests cover request translation and response normalization.
- Error mapping is tested (e.g. rate limit, auth).

acceptance_criteria:
- Adapter correctly wraps Anthropic SDK and translates formats without leaking abstractions.

## 3. Implement Gemini Native Adapter

type: implementation

files:
- `internal/providers/gemini/adapter.go`
- `internal/providers/gemini/adapter_test.go`

action:
- Import `google.golang.org/genai`.
- Implement `providers.ProviderAdapter`.
- Map messages to Gemini `Content` / `Part` objects. Handle `system` role separately as per Gemini SDK instructions (usually mapped to `SystemInstruction`).
- Forward `Temperature` and `MaxTokens` via `GenerateContentConfig`.
- Map Gemini response to `llm.LLMResponse`.
  - Finish reason mapping: normal stop -> `stop`, max tokens -> `length`.
- Map SDK errors to structured gateway errors.

verify:
- Unit tests cover request translation, system prompt extraction, and response normalization.

acceptance_criteria:
- Adapter correctly interfaces with Gemini API and normalizes responses to the shared internal structure.
