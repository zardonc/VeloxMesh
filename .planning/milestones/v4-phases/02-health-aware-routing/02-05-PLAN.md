# Phase 2.5 Plan: Provider Retry and Fallback Execution

<objective>
Add deterministic request-level retry/fallback execution for non-streaming chat completions.

The phase is complete when a normal `/v1/chat/completions` request can recover from a retryable provider failure by trying another healthy/degraded provider, while preserving strict `X-Route-To` behavior, avoiding repeated providers within one request, updating provider health per attempt, exposing safe fallback metadata, and keeping successful responses OpenAI-compatible.
</objective>

<context>
Use `.planning/phases/02-health-aware-routing/02-05-CONTEXT.md` as the source of phase decisions.

Important locked decisions:
- D-01 through D-04: Implement provider retry/fallback execution as a small in-process data-plane feature; do not build full circuit breakers.
- D-05 through D-08: Retry only non-streaming requests and only retry transient/provider-side categories.
- D-09 through D-13: Fallback should not repeat providers, must respect health, must preserve strict `X-Route-To`, and must not perform model substitution.
- D-14 through D-16: Use conservative static config, default max attempts of 2, and avoid same-provider retry/backoff in this phase.
- D-17 through D-20: Preserve OpenAI-compatible success responses, expose safe attempt metadata, record per-attempt diagnostics, and update health for every attempted provider.
</context>

<scope>
## Must Build

- Retryability policy based on Phase 2.4 provider error categories.
- Static retry/fallback configuration with conservative defaults.
- Exclusion-aware provider selection for request-scoped fallback.
- Gateway service attempt loop for non-streaming `/v1/chat/completions`.
- Strict no-fallback behavior for `X-Route-To`.
- Per-attempt health updates, metrics, and sanitized diagnostics.
- Safe response headers/metadata for attempt count and fallback use.
- Unit and integration tests for fallback success, fallback exhaustion, non-retryable errors, strict override behavior, and health updates.

## Must Not Build

- Redis, PostgreSQL, Admin API, runtime retry policy CRUD, or config hot reload.
- Full circuit breaker state machine.
- Same-provider retry with exponential backoff.
- Streaming/SSE retry.
- Hedged/parallel requests.
- Model degradation or model substitution.
- Cost-aware routing/fallback, semantic cache, or rate limiting.
</scope>

<truths>
- Successful `/v1/chat/completions` responses must remain OpenAI-compatible.
- `X-Route-To` remains strict: if a client asks for one provider, do not silently route to another.
- Provider attempts must not leak API keys, raw prompts, auth headers, raw upstream bodies, or full upstream error payloads.
- Health updates must happen for every attempted provider, not only the final provider.
- Tests must use fake upstream servers and must not call real LLM providers.
- Avoid sleeps/backoff in tests; keep fallback deterministic.
</truths>

type: implementation

files:

- `internal/config/config_test.go`

- `internal/errors/errors_test.go`

action:
- Add a small retry/fallback config shape to `config.Config`, for example:
  - `FallbackEnabled bool`
  - `MaxAttempts int`
  - optionally `FallbackBackoff string` only if the implementation can keep default behavior at zero/no sleep.
- Parse this config from JSON config files, with conservative defaults:
  - enabled when there is more than one configured provider, unless explicitly disabled.
  - max attempts default: 2.
  - max attempts minimum: 1.
  - max attempts cannot exceed a safe static upper bound such as 5.

- Mark non-retryable:
  - `provider_invalid_request`
  - `provider_invalid_model`
  - `provider_auth_error`
  - routing errors and admission errors.
- Treat `context.Canceled` and `context.DeadlineExceeded` as stop-now conditions for further attempts.

## 3. Implement Gateway Attempt Loop

type: implementation

files:
- `internal/gateway/service.go`
- `internal/gateway/service_test.go`
- `internal/llm/types.go`

action:
- Refactor provider call handling in `HandleChatCompletion` into a per-attempt helper so health and metrics are updated consistently for each attempt.
- For requests without `RouteOverride`, fallback is enabled, and `MaxAttempts > 1`:
  - select provider.
  - call admission for that selected provider.
  - begin health request.
  - call adapter.
  - end health request with health-impact-adjusted error.
  - record metrics/outcome.
  - if success: return response with final provider, strategy, attempt count, and fallback metadata.
  - if error is non-retryable: return immediately.
  - if context is canceled/deadline exceeded: return immediately.
  - if retryable: add provider to attempted set and select another eligible provider until attempts are exhausted.
- For requests with `RouteOverride`, preserve current single-attempt behavior.
- Avoid retrying the same provider within a single request.
- Decide final error behavior when all attempts fail:
  - Prefer returning the last structured provider error while preserving health/metrics for all attempts.
  - If selection exhausts candidates after retryable failure, return the last provider error rather than hiding it behind `no_healthy_provider`, unless no provider was ever attempted.
- Keep admission release handling correct on every attempt.

verify:
- Unit-test success on first attempt.
- Unit-test retryable first attempt then fallback success.
- Unit-test retryable failures until max attempts returns structured error.
- Unit-test invalid request/auth/model errors do not fallback.
- Unit-test `RouteOverride` failure does not fallback.
- Unit-test context cancellation stops attempts.
- Unit-test per-attempt health updates and pending decrements.

acceptance_criteria:
- The attempt loop is explicit and testable.
- Fallback does not alter successful OpenAI-compatible response bodies.
- Health and admission cleanup are correct on every path.
