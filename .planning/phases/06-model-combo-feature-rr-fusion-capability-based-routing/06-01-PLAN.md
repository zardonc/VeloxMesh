# Phase 6 Plan 01: Persistent Combo Models and Routing

## Objective
- **What:** Add persisted model combos, expose them as model names, and route chat completions through round-robin, fusion, capacity auto-switch, and ordered fallback.
- **Why:** Phase 6 makes the gateway choose across multiple connected models while preserving the OpenAI-compatible client contract.
- **Output:** Durable combo configuration, admin combo endpoints, combo-aware `/v1/models`, combo chat routing, fusion judge synthesis, and integration coverage.

- [ ] Combos persist in SQLite control state; retained PostgreSQL code remains passing but is not expanded in Phase 06.
- [ ] Admin APIs can create, read, update, list, and delete combos with schema validation.
- [ ] `/v1/models` includes enabled combo names.
- [ ] `POST /v1/chat/completions` accepts combo names as `model`.
- [ ] Combo routing invokes providers with the selected upstream model, not the combo name.
- [ ] Round-robin combo routing chooses healthy eligible members and ordered fallback tries later members after retryable failures.
- [ ] Capacity auto-switch routes image/PDF/audio/tool requests to capable members first.
- [ ] Fusion performs N panel calls plus one configured judge call and returns one OpenAI-compatible response.
- [ ] Usage settlement reflects the actual upstream calls, including N+1 fusion billing.
