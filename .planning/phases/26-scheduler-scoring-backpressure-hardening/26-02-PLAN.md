<scope_review>

## Current Code Findings

- Bug 1: `SynchronousRunner.waitForTask` already waits for registry completion on `ErrQueueEmpty` and no longer uses `ResultRegistry.IsRunning` as a failure gate.
- Bug 2: `SchedulerRolloutController.RecordAlert` already truncates alerts to the latest 100 entries.
- Bug 3: `PredictionQualityRecorder.recordMAPEAlert` and `recordErrorSpikeAlert` still trigger alerts from single samples.

## Chosen Design

- Keep alert windows in memory inside `PredictionQualityRecorder`; durable rollups remain unchanged.
- Use a small ring buffer of ONNX quality samples.
- Default window size: 100.
- Admin runtime knob: `quality_sample_window`.
- Config/env/json default: `scheduler.quality_sample_window` / `SCHEDULER_QUALITY_SAMPLE_WINDOW`.
- Minimum samples: derive as `min(20, quality_sample_window)` to avoid early 1/1 false positives without adding another knob.
- Alert on threshold crossing, not on every sample while already degraded.
- The recorder must observe runtime `quality_sample_window` changes without process restart; if the window shrinks, drop oldest samples beyond the new cap.

## Implementation Complexity and Impact

- Complexity: medium. The change is local to scheduler quality recording plus config/admin surfaces, but it touches several tests and public operator examples.
- Runtime impact: bounded memory only. The default window stores 100 small samples per scheduler process; larger admin values must remain bounded by validation.
- Behavioral impact: quality alerts become less sensitive to one-off ONNX failures and may alert later, after the derived minimum sample count is reached.
- Compatibility impact: existing rollout alert JSON gains `quality_sample_window`; admin PATCH must remain backward compatible for `onnx_rollout_percent`.

## Out of Scope

- Persisting alert windows across process restart.
- Adding automatic ONNX rollout rollback.
- Adding time-window buckets; this plan uses sample-count windows only.
