<scope>
## Must Build

- Explicit config schema structs or grouped helpers for provider identity, provider type, endpoint/auth, model defaults, timeout, health-check overrides, fallback/retry settings, and capability overrides only if needed for current Phase 2.7 code.
- Secret-safe validation helpers for:
  - duplicate and empty provider IDs.
  - unknown provider types.
  - missing or invalid base URLs.
  - empty models.
  - `default_model` values that are not present in `models`.
  - invalid or negative durations.
  - invalid health-check thresholds and max concurrency.
  - fallback attempts that exceed configured provider count or conflict with disabled fallback.
- Backward-compatible loading for env-only single-provider OpenAI-compatible config.
- Optional auth reference shape that avoids docs normalizing raw secrets while preserving the existing `api_key` JSON field for compatibility.
- README config docs with a secret-safe example covering OpenAI-compatible, Anthropic, and Gemini providers.
- Tests for validation behavior and secret-safe error text.

- Database-backed provider config.
- Secret manager integration.
- Runtime config hot reload or CRUD.
- Redis or PostgreSQL.
- Streaming, rate limiting, semantic cache, cost governance, or live model discovery.
- Provider-native request/response mapping in common config code.
- New routing behavior through capability overrides.
</scope>

## 1. Refactor Static Config Schema and Validators

type: implementation

files:
- `internal/config/config.go`
- `internal/app/app.go`
- `internal/health/prober.go`

action:
- In `internal/config/config.go`, refactor the current organic config definitions into explicit schema concepts while preserving existing JSON compatibility per D-01, D-02, D-03, D-05, D-06, D-07, D-08, D-09, D-10, D-11, and D-12.
- Keep `ProviderConfig` compatible with existing fields: `id`, `type`, `base_url`, `api_key`, `models`, `default_model`, `timeout`, `weight`, and `health_check`.
- Add an auth reference shape, for example `ProviderAuthConfig` under `auth`, that supports an environment-variable reference such as `api_key_env`. Preserve existing `api_key` for compatibility; do not add secret-manager lookup or external credential storage.
- Add helper methods or load-time resolution so `internal/app/app.go` can pass the resolved provider API key to adapters without knowing whether it came from legacy `api_key`, env fallback, or `auth.api_key_env`.
- Do not move OpenAI, Anthropic, or Gemini request/response mapping into config code per D-04.
- Validate base URLs with `net/url`: scheme must be `http` or `https`, host must be non-empty, and validation errors should identify `provider <id> base_url` without echoing sensitive config values.
- Validate model defaults: if `default_model` is set, it must match an entry in `models`; model lists must not be empty.
- Replace silent `MaxAttempts` clamping with explicit fallback validation per D-11 and D-13:
  - default zero values explicitly based on provider count and fallback state.
  - reject values less than 1.
  - reject fallback-enabled `max_attempts` greater than configured provider count.
  - reject explicit multi-attempt fallback settings when fallback is disabled.
- Extract deterministic helpers per D-15, such as `applyDefaults`, `validateProvider`, `validateProviderBaseURL`, `validateProviderModels`, `validateFallback`, `validateHealthCheckConfig`, and `validateDurationField`.
- Keep Phase 2.6 health-check config names intact and validate global interval, timeout, initial delay, stale-after, thresholds, and max concurrency; validate provider override durations and thresholds that the prober consumes per D-10 and D-13.
- Add capability override schema only if the existing Phase 2.7 capability path requires static overrides for safe config validation; if added, keep it provider-neutral and do not route on it per D-12.
- Update `internal/app/app.go` only where needed to consume the resolved auth value or renamed schema helpers.
- Update `internal/health/prober.go` only where needed to consume any renamed health-check accessors; keep probe behavior unchanged.

verify:
- `go test ./internal/config`
- `go test ./internal/app ./internal/health`

acceptance_criteria:
- Config schema clearly represents Phase 2.8 fields without breaking current JSON or env fallback behavior.
- Validation is helper-driven, deterministic, and secret-safe.
- App wiring still selects adapters by provider type and keeps provider-native behavior in adapter packages.

  - missing base URL.

  - empty model list.
  - `default_model` mismatch.
  - invalid provider timeout.
  - negative provider timeout.
  - invalid global health-check duration.
  - invalid provider health-check override duration.
  - invalid health-check thresholds.
  - invalid max concurrency.
  - fallback `max_attempts` greater than provider count.
  - explicit multi-attempt setting when fallback is disabled.
