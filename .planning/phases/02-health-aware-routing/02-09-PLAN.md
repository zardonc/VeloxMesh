# Phase 2.9 Plan: Provider Model Catalog and Routing Eligibility

<objective>
Build an internal model catalog that maps configured model IDs to eligible providers and provider-neutral capabilities, then route chat completion requests through that single eligibility source before health and strategy selection.

This plan is complete when catalog APIs are copy-safe, router selection excludes model or capability-ineligible providers, strict overrides enforce eligibility, fallback never attempts ineligible providers, `/v1/models` remains OpenAI-compatible, and full Go verification passes.
</objective>

## Must Build

- Internal model catalog derived from static provider config, adapter model lists, and adapter capability metadata per D-01, D-02, D-05, D-09, D-10, D-11, and D-12.
- Catalog metadata for model ID, provider ID, provider type, provider default-model status, and copied provider-neutral capabilities per D-05 and D-08.
- Shared-model representation where one model ID can have multiple eligible providers per D-06 and D-22.
- Provider-specific model eligibility where a configured model is eligible only for providers that expose it per D-07 and D-23.
- Router integration that filters by model and `chat_completions` eligibility before health/latency strategy selection per D-14, D-15, and D-18.
- Strict override eligibility checks per D-16.
- Fallback eligibility checks using the same catalog source per D-17.
- Deterministic unknown-model and no-eligible-provider structured errors before provider execution per D-13 and D-24.
- Tests for shared model names, provider-specific models, unknown models, capability-ineligible providers, strict override eligibility, fallback eligibility, copy safety, and public model response compatibility per D-22, D-23, D-24, D-25, D-26, and D-27.

## Must Not Build

- Live model discovery from provider APIs per D-03.
- Model aliasing, substitution, degradation, rewriting, or fallback model selection per D-04.
- Cost-aware routing.
- Embeddings, images, multimodal, streaming, or tool/function calling API implementation.
- Admin API or Admin Console provider/model CRUD.
- PostgreSQL, Redis, database-backed catalog state, runtime hot reload, rate limiting, semantic cache, cost governance, or observability exporters.
- Provider-native request/response mapping outside adapter packages.
</scope>

<truths>
- The catalog is internal; public clients still see OpenAI-compatible `/v1/models` JSON only.
- `providers.Registry` already preserves stable provider ordering through `ids` and already exposes copied capability metadata.
- Phase 2.8 config supplies provider IDs, provider types, model lists, and default model values; Phase 2.9 should consume that data rather than creating a new config source.
- `routing.HealthAwareRouter` is the selection boundary where eligibility must happen before health filtering and strategy selection.
- `gateway.Service` fallback already calls `SelectExcluding`; once the router filters eligibility correctly, fallback naturally avoids ineligible providers.
- Unknown or unsupported models are request eligibility failures, not provider health failures.
- Logs, errors, tests, and readiness must not expose API keys, authorization headers, raw prompts, raw upstream bodies, or sensitive provider payloads per D-21.
</truths>

## 1. Add Copy-Safe Provider Model Catalog APIs

type: implementation

files:
- `internal/providers/catalog.go`
- `internal/providers/capabilities.go`
- `internal/providers/registry.go`
- `internal/providers/registry_test.go`

action:
- Create `internal/providers/catalog.go` with provider-neutral catalog types and constructors per D-01, D-02, D-05, D-06, D-07, D-08, D-09, D-10, D-11, and D-12.
- Define catalog-facing types such as `ModelCatalog`, `ModelEntry`, and `ModelProvider` or equivalent names that expose model ID, provider ID, provider type, `DefaultModel` bool, and copied `CapabilitySet`.
- Build the catalog from registered adapters plus the Phase 2.8 static config provider records passed into `NewRegistry`; use adapter `Models()` and `Capabilities()` as runtime truth, and use config `DefaultModel` only to mark default-model status.
- Preserve registry provider order when constructing each model's eligible provider list. Shared model IDs should appear once in `ListModels()` while retaining multiple eligible providers internally per D-06 and D-22.
- Keep provider-specific model IDs eligible only for the providers whose adapter model list contains them per D-07 and D-23.
- Add capability helpers in `internal/providers/capabilities.go`, such as `SupportsOperation(Operation) bool`, if that keeps eligibility logic deterministic and table-testable.
- Add registry APIs that delegate to the catalog, for example `ModelCatalog() ModelCatalog`, `GetAllModels() []string`, `EligibleProviders(model string, operation Operation) []ModelProvider`, `ProviderSupports(providerID string, model string, operation Operation) bool`, and `DefaultModel(providerID string) (string, bool)`. Use exact names that fit existing Go style, but keep the behaviors explicit.
- Return defensive copies from all catalog and registry APIs: model slices, provider slices, and capability slices/maps must not be caller-mutable per D-08.
- Do not call provider APIs, do not infer live provider model availability, and do not rewrite client model IDs per D-03 and D-04.

verify:
- `go test ./internal/providers -run "Registry|Catalog|Model|Capability|Copy"`
- `go test ./internal/providers`

acceptance_criteria:
- Registry callers can list all model IDs, list eligible providers for a model and `chat_completions`, check provider eligibility, and read default-model metadata from one copy-safe catalog source.
- Tests cover shared model names, provider-specific model names, capability-ineligible providers, stable order, default-model flags, unknown models returning no eligible providers, and defensive-copy behavior.

## 2. Enforce Catalog Eligibility in Router Selection

type: implementation

files:
- `internal/errors/errors.go`
- `internal/routing/router.go`
- `internal/routing/router_test.go`
- `internal/gateway/service.go`
- `internal/gateway/service_test.go`

action:
- Add structured eligibility errors in `internal/errors/errors.go` using existing `GatewayError` style per D-13 and D-21:
  - `no_eligible_provider` with HTTP 400 for known request shapes where no configured provider supports the requested model and operation.
  - `ineligible_provider_override` with HTTP 400 when `X-Route-To` names a configured provider that does not support the requested model and operation.
- Keep error messages generic and secret-safe. Do not include API keys, authorization headers, raw prompts, raw upstream bodies, or provider-native payload details per D-21.
- Update `routing.Router` and `HealthAwareRouter` so `Select` and `SelectExcluding` use catalog eligibility for `providers.OperationChatCompletions` before applying exclusions, health filtering, round-robin, least-latency, or cold-start fallback per D-14, D-15, and D-18.
- For non-override routing, compute eligible providers for `req.Model` and `chat_completions`, remove excluded provider IDs, then apply existing health filtering and strategy selection. If the requested model has no eligible providers before health filtering, return `ErrNoEligibleProvider`; if eligible providers exist but are all unhealthy, preserve `ErrNoHealthyProvider`.
- For strict override routing, preserve current strict behavior while enforcing deterministic order per D-16: unknown provider override returns `ErrUnknownProviderOverride`; known but model/operation-ineligible provider returns `ErrIneligibleProviderOverride`; known and eligible but unhealthy provider returns `ErrUnhealthyProviderOverride`.
- Preserve fallback behavior in `gateway.Service`: `X-Route-To` disables fallback, retryable provider failures add the failed provider to `attempted`, and subsequent `SelectExcluding` calls can only return catalog-eligible providers per D-17.
- Do not implement model substitution, provider degradation, cost-aware routing, streaming, embeddings/images, tools, or multimodal behavior per D-03 and D-04.

verify:
- `go test ./internal/routing -run "Select|Eligibility|Override|Fallback|Latency|RoundRobin"`
- `go test ./internal/gateway -run "HandleChatCompletion|Fallback|Eligibility"`
- `go test ./internal/routing ./internal/gateway`

acceptance_criteria:
- Unknown requested models fail before provider execution.
- Capability-ineligible providers are excluded before health and strategy selection.
- Strict override fails for ineligible providers and does not fallback.
- Fallback excludes both previously failed providers and catalog-ineligible providers.
- Existing round-robin, least-latency, cold-start, unhealthy skip, and override health behavior remains intact after eligibility filtering.
