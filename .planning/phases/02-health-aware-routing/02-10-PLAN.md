<objective>
Create a reusable Go conformance test harness that every current and future provider adapter can run against the provider-neutral `providers.ProviderAdapter` contract.

<interfaces>
`providers.ProviderAdapter` is the contract under test:
- `ID() string`
- `Models() []string`
- `Complete(ctx context.Context, req *llm.LLMRequest) (*llm.LLMResponse, error)`
- `HealthCheck(ctx context.Context) providers.HealthStatus`
- `Capabilities() providers.CapabilitySet`

`providers.CapabilitySet` is the provider-neutral metadata surface:
- `ProviderType providers.ProviderType`
- `SupportedOperations []providers.Operation`
- `InputModalities []providers.Modality`
- `OutputModalities []providers.Modality`
- `Streaming bool`
- `ToolCalling bool`
- `GenerationParameters []providers.GenerationParameter`

`errors.GatewayError` is the structured provider error contract:
- `Code string`
- `Message string`
- `HTTPStatus int`

`llm.LLMRequest` and `llm.LLMResponse` are the provider-neutral request and response shapes adapters must consume and return.
</interfaces>

## 1. Add Provider-Neutral Adapter Test Harness

type: implementation

files:
- `internal/providers/adaptertest/harness.go`
- `internal/providers/adaptertest/harness_test.go`
- `internal/providers/adaptertest/README.md`

action:
- Create `internal/providers/adaptertest` as the shared test-support package per D-01 and D-06. Production code must not import it; only adapter test files and the helper package itself may reference it.
- Define a compact `ConformanceSpec` or equivalently named type per D-07. It should describe the adapter under test, expected provider ID, configured models, expected `providers.CapabilitySet`, optional defensive-copy checks, health cases, completion success cases, finish-reason cases, provider error cases, request-inspection hooks, and forbidden secret substrings.
- Keep the harness provider-neutral: it may import `testing`, `context`, `strings`, `veloxmesh/internal/errors`, `veloxmesh/internal/llm`, and `veloxmesh/internal/providers`, but it must not import `openai`, `anthropic`, `gemini`, SDK packages, gateway handlers, routing, config, or app wiring.
- Provide harness functions such as `RunConformance(t, spec)`, `AssertGatewayError(t, err, code)`, `AssertSecretSafeError(t, err, forbiddenSubstrings)`, and focused helpers for capability equality, model copy checks, health status checks, response shape checks, and finish reason checks. Exact names can follow local Go style.
- Make provider-specific setup injectable per D-10. Adapter packages should own `httptest.Server` setup, fake SDK base URLs, native response payloads, and request inspection, then pass provider-neutral expected outcomes into the harness.
- Ensure request mapping sanity checks can assert model, system/user/assistant messages, temperature, and max tokens without the harness knowing provider-native wire formats per D-12.
- Ensure structured error cases cover auth, rate limit, invalid request, invalid model, timeout, unavailable, and bad response only where the adapter package can produce them deterministically without external calls per D-15.
- Add helper self-tests in `harness_test.go` for secret-safety assertions and capability/model comparison behavior.
- Start `internal/providers/adaptertest/README.md` with the future-adapter contract: create fake upstream behavior, instantiate `ConformanceSpec`, call `RunConformance`, keep native SDK tests beside the adapter, and register the adapter for routing only after the harness passes per D-05 and D-23.

verify:
- `go test ./internal/providers/adaptertest`

acceptance_criteria:
- A reusable provider-neutral harness package exists.
- Harness APIs are table-driven and deterministic.
- Harness helper tests pass.
- The helper package has no provider-specific, routing, handler, config, or app wiring imports.
- The authoring guide explains the test contract for future adapters.
