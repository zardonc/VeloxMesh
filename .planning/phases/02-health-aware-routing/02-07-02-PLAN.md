# Phase 2.7 Plan 2: Provider-Agnostic Capability Exposure

<objective>
Expose provider capability metadata through provider-agnostic gateway boundaries and secret-safe operational output.

This plan is complete when routing/gateway service layers can provide capability metadata from the registry, `/readyz` can optionally show safe capability summaries, and `/v1/models` remains OpenAI-compatible.
</objective>

<context>
Use `.planning/phases/02-health-aware-routing/02-07-CONTEXT.md` and `.planning/phases/02-health-aware-routing/02-07-01-PLAN.md`.

This plan depends on Plan 1's capability contract and registry metadata APIs. Do not start this plan until Plan 1 compiles and passes `go test ./...`.
</context>

<scope>
## Must Build

- Provider-neutral capability accessors through routing and/or gateway service boundaries.
- Secret-safe readiness capability summaries if they stay compact.
- Tests proving HTTP handlers remain provider-agnostic.
- Tests proving `/readyz` does not expose secrets.
- Tests proving `/v1/models` response shape remains OpenAI-compatible.

## Must Not Build

- New adapter capability contract work; that belongs to Plan 1.
- Streaming implementation.
- Tool/function calling implementation.
- Multimodal implementation.
- Live model discovery.
- Admin API or Admin Console.
- Redis, PostgreSQL, runtime CRUD, rate limiting, semantic cache, or cost governance.
</scope>

<truths>
- `/readyz` is operational metadata and may include provider-neutral capability summaries.
- `/v1/models` is client-facing OpenAI-compatible API and must not add non-standard fields to model items.
- HTTP handlers must not import `internal/providers/openai`, `internal/providers/anthropic`, or `internal/providers/gemini`.
- Capability output must not contain API keys, authorization headers, base URLs, raw prompts, raw upstream bodies, raw errors, or SDK-native type names.
</truths>

## 1. Add Gateway Capability Accessors

type: implementation

files:
- `internal/routing/router.go`
- `internal/routing/router_test.go`
- `internal/gateway/service.go`
- `internal/gateway/service_test.go`

action:
- Add provider-neutral capability access through the existing routing and gateway service boundaries, for example:
  - `Router.GetProviderCapabilities() []providers.ProviderCapabilities`.
  - `Service.GetProviderCapabilities() []providers.ProviderCapabilities`.

verify:
- `go test ./internal/routing ./internal/gateway`
- Tests prove accessors return deterministic provider order and copy-safe metadata.

acceptance_criteria:

## 2. Add Secret-Safe Readiness Capability Summaries

type: implementation

files:
- `internal/http/handlers/health.go`
- `tests/integration/health_test.go`

action:
- Update `/readyz` provider details to include compact capability summaries where available:
  - provider type.
  - operations.
  - input modalities.
  - output modalities.
  - streaming support flag.
  - tool/function calling support flag.
  - generation parameters.
  - configured numeric constraints if Plan 1 includes them.
- Keep existing provider health/probe readiness fields.
- Do not include API keys, authorization headers, base URLs, raw prompts, raw provider payloads, raw provider errors, or SDK-native type names.
- If capability summaries make readiness too noisy, expose only provider type plus high-level support flags and document the choice in tests/comments.

verify:
- `go test ./tests/integration -run Ready`
- Integration tests confirm readiness includes safe capability metadata.
- Integration tests confirm readiness output does not contain configured API key values or base URLs.

acceptance_criteria:
- Operators can see what each provider supports at a high level from readiness without seeing sensitive provider data.
