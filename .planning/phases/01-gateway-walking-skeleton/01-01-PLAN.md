# Phase 1 Plan: Gateway Walking Skeleton

<objective>
Create the VeloxMesh project foundation and prove the smallest useful gateway loop:

client -> VeloxMesh HTTP API -> request validation -> gateway core -> OpenAI-compatible provider adapter -> LLM provider -> normalized response -> client.

The phase is complete only when a local client can call `POST /v1/chat/completions` through VeloxMesh and receive either a normalized provider response or a normalized gateway error with request logging.
</objective>

<scope>
## Must Build In Phase 1

- A runnable backend service project.
- Minimal HTTP API surface:
  - `GET /health`
  - `POST /v1/chat/completions`
- OpenAI-compatible request/response shape for the chat endpoint.
- One provider adapter: `openai-compatible`.
- Provider registry that selects the default provider from config.
- Environment-based configuration.
- Structured logging with request id and latency.
- Centralized error mapping.
- Focused tests for health, validation, gateway orchestration, provider success, and provider failure.
- README instructions for local setup and curl-based verification.

## Explicitly Deferred

- Multi-provider routing policies.
- Provider fallback, retries, circuit breakers, and load balancing.
- Streaming/SSE.
- API key authentication and tenant management.
- Rate limiting.
- Semantic cache.
- Usage metering, billing, and cost controls.
- Agent workflow orchestration.
- Dashboard or frontend UI.
- Database persistence.
- Distributed tracing backend integration.
</scope>

<truths>
- The first phase optimizes for a real end-to-end call path, not breadth.
- OpenAI-compatible API shape is the default compatibility target.
- Provider-specific behavior must stay behind a provider adapter.
- Configuration must be loaded from environment variables and validated at startup.
- Logs must not print API keys, tokens, full prompts, or sensitive headers.
- Errors returned to clients must be stable and structured.
</truths>
