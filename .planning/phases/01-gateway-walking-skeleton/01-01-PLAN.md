# Phase 1 Plan: Go Gateway Walking Skeleton

<objective>
Create the Velox AI Gateway foundation using the architecture document as the source of truth:

client -> Go Chi data-plane API -> request id/logging/auth middleware -> OpenAI-compatible request normalization -> gateway service -> routing boundary -> admission boundary -> provider adapter -> upstream LLM provider -> normalized OpenAI-compatible response -> client.

The phase is complete when a local client can call `POST /v1/chat/completions` against the Go gateway and receive either a normalized provider response or a structured gateway error, with request id propagation and latency-aware logs.
</objective>

<architecture_source>
Canonical design document:

`C:\Users\inthe\IdeaProjects\Notes-sur-l-IA\Projects\Agent-gateway\gateway-architecture.md`

Locked architecture decisions from that document:

- Gateway binary uses Go 1.22+ and Chi router.
- Gateway is a single Go process with logical control-plane and data-plane layers.
- Base forwarding path is latency-sensitive; optional features must not pollute the hot path.
- Public data-plane API is OpenAI-compatible JSON over HTTP/SSE.
- Provider adapters, routing strategies, and middleware are Go interface-based.
- Logs use Go `slog` structured JSON.
- Future storage/control dependencies are PostgreSQL 16 + pgvector and two Redis instances.
- Semantic cache, full admin console, cost governance, and advanced routing are later phases.
</architecture_source>

<scope>
## Must Build In Phase 1

- Go module and runnable gateway binary.
- Chi-based HTTP router with standard `net/http` middleware chain.
- Data-plane endpoints:
  - `GET /healthz`
  - `GET /readyz`
  - `POST /v1/chat/completions`
- Minimal OpenAI-compatible chat request/response structs.
- Internal `LLMRequest` / `LLMResponse` normalization layer.
- Provider adapter interface matching the architecture direction:
  - `ID() string`
  - `Models() []string`
  - `Complete(ctx, *LLMRequest) (*LLMResponse, error)`
  - `HealthCheck(ctx) HealthStatus`
- One concrete OpenAI-compatible provider adapter.
- Provider registry with static config loaded from environment/config file.
- Routing boundary with a first simple strategy, preferably single-provider or least-latency stub.
- Admission-control boundary with pass-through implementation and metrics/log hooks.
- Lightweight auth middleware preserving the architecture contract:
  - require `Authorization: Bearer ...`
  - support one bootstrap/dev API key from environment for Phase 1
  - leave PostgreSQL/Redis-backed key verification for later.
- Structured errors and response headers:
  - `X-Request-ID`
  - `X-Provider`
  - `X-Model`
  - `X-Latency-E2E-Ms`
- `slog` JSON logs with request id, route, status, latency, provider, and model.
- Focused Go tests for health, auth, chat validation, gateway orchestration, and provider failure.
- README instructions for local verification.

## Explicitly Deferred

- PostgreSQL schema and migrations.
- Redis state/cache integration.
- Real API key persistence and Redis auth cache.
- Sliding-window rate limiting.
- Full admission queueing and priority scheduling.
- Composite-score routing.
- Circuit breaker and fallback chain.
- SSE streaming proxy.
- `/v1/completions`, `/v1/embeddings`, and `/v1/models`.
- Admin API.
- Semantic cache and Python embedding sidecar.
- Prometheus `/metrics` endpoint beyond minimal internal counters.
- OpenTelemetry tracing.
- Cost governance and usage aggregation.
- Docker Compose stack.
- Next.js Admin Console.
</scope>

<truths>
- Do not use TypeScript, Node.js, Fastify, Hono, or Vitest for the gateway.
- Phase 1 is Go-first and must keep the future architecture visible through package boundaries and interfaces.
- The hot path must be simple and easy to benchmark.
- Provider-specific code belongs behind adapters.
- Routing, admission control, auth, and observability should exist as thin boundaries even if their Phase 1 implementation is minimal.
- Phase 1 may use static environment/config data where the architecture later uses PostgreSQL and Redis.
- Secrets and raw prompts must not be logged.
</truths>
