Wire durable provider state into runtime provider activation, routing, readiness, and model listing without adding Admin API routes yet.

Output: runtime provider manager, actionable missing-configuration errors, disabled-provider filtering, and tests proving reload without process restart.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| durable storage -> runtime adapters | Encrypted provider records become live adapter instances after decryption. |
| Admin/runtime reload -> data plane | Runtime provider state changes affect routing and model listing. |
| disabled provider -> router | Disabled records must not receive traffic. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Information Disclosure | runtime loader errors | mitigate | Missing-secret and activation errors name provider IDs/fields only and never include API keys, decrypted secrets, ciphertext, raw prompts, or provider payloads per D-03 and D-10. |
| T-03-03-02 | Tampering | runtime activation | mitigate | Publish a complete runtime snapshot only after validation succeeds; failed activation leaves previous snapshot unchanged per D-22. |
| T-03-03-03 | Denial of Service | empty provider config | mitigate | Return actionable `no_active_provider_config` instead of panics or ambiguous no-provider behavior per D-03. |
| T-03-03-04 | Elevation of Privilege | disabled providers | mitigate | Filter disabled providers before registry/catalog/routing so disabled records cannot receive traffic per D-19. |
| T-03-03-SC | Tampering | package installs | accept | This plan adds no package-manager dependencies. |
</threat_model>
