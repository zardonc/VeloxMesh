## Cross-Phase Integration Issues

### 1. Real app path does not publish replication events

`internal/app/app.go:244` creates `replication.NewConsumer(...)` and `internal/app/app.go:248` creates `NewRecoveryWorker(...)`. No app wiring calls `replication.NewRedisStreamProducer(...)` or wraps the opened repository with `replication.NewRepository(...)`.

Impact: leader writes made through the real application will update SQLite locally, but the Redis Stream producer/decorator path is not active in production app startup. This blocks REPL-01 and weakens RECOV-01.

### 2. Stream name mismatch

The app consumer listens on `controlstate:events` at `internal/app/app.go:244`, while Phase 12 summaries claim `veloxmesh:control:stream`.

Impact: even if a producer is later wired, producer and consumer must share the same stream naming contract or replicas will not receive changes.

### 3. WAL lag pending/distance is not implemented

`internal/controlstate/replication/consumer.go:41` implements `ReportLag`, but `consumer.go:47` comments that pending should use XPENDING and `consumer.go:52` always returns `Pending: 0`.

Impact: `/readyz` and `/admin/v1/topology` can never report Redis Stream pending distance accurately, so stale replicas may look healthy.

### 4. Redis outage scenario is weaker than the milestone definition

`tests/integration/multinode_coordination_test.go:45` is named `TestMultiNodeRedisOutage`, but lines 51-67 fast-forward lease expiration and then switch to follower write rejection. It does not fully simulate Redis/network disruption, failed sync recording, and recovery replay.

Impact: TEST-01 remains partial.
